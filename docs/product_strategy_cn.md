# 产品、市场与交互建议

## 1. 先给出我的核心判断

你现在最大的困惑不是“技术还能做什么”，而是：

> **这套能力到底应该被包装成什么产品。**

我的判断是：

> **最适合当前代码现实的产品定义，是“面向中学理科教师的 AI 阅卷与复核工作台”。**

这一定义比“教育 AI 平台”“学生端 AI 应用”“学校级系统”都更贴近你已经完成的东西。

---

## 2. 为什么我建议这样定义

因为你现在最强的不是：

1. 面向学生的持续陪练体验
2. 校级教务协同
3. 通用文档理解

而是：

1. 参考答案生成 rubric
2. 批量上传学生作答
3. 自动评分
4. 低置信度与异常样本复核
5. 证据化报告
6. 反馈沉淀与标注资产积累

这些能力天然指向的是**教师工作流提效产品**。

---

## 3. 推荐市场定位

## 3.1 推荐的一句话定位

> **帮助理科老师批量阅卷、快速复核、自动生成讲评依据的 AI 工作台**

## 3.2 推荐的首批用户

1. 中学物理老师
2. 中学数学老师
3. 小型教培机构的批改老师 / 教研老师
4. 校内备课组、年级组

## 3.3 他们真正在意什么

不是“你用了什么模型”，而是：

1. 这次作业我能不能更快改完？
2. 哪些卷子机器不确定，需要我看？
3. 这题学生最常错在哪里？
4. 我能不能少做重复劳动？
5. 能不能直接辅助我讲评？

---

## 4. 不建议当前就追的方向

1. 学生个人学习助手
2. 全学科覆盖
3. 学校行政/教务系统
4. 通用图文理解平台
5. “万能教育 AI”

原因很简单：  
这些方向都不是你当前代码最强、最容易落地变现的方向。

---

## 5. 产品应该围绕哪些核心场景

## 场景 A：一次小测的批量阅卷

### 教师目标

有标准答案，想一次处理一个班或一组学生作答。

### 理想交互

1. 创建任务
2. 上传参考答案
3. 上传学生作答
4. 选择评分规则
5. 提交并实时查看进度
6. 查看班级汇总
7. 只复核机器不确定的样本

### 你当前系统已经能支撑的部分

1. 上传
2. 异步任务
3. 批量处理
4. 状态查询
5. 报告读取
6. 待复核链路

### 还缺什么

1. 班级汇总页
2. 复核优先级排序
3. 题目级错误统计
4. 更好用的导出结果

---

## 场景 B：单题讲评准备

### 教师目标

老师不是只想知道“谁错了”，而是想知道：

1. 哪一步最容易错
2. 常见错因是什么
3. 哪些学生是计算错误，哪些是概念错误
4. 讲评课上应该重点讲什么

### 应该产出的不是单人报告，而是“题目级信息”

推荐页面结构：

1. **题目总览卡**
   - 提交人数
   - 正确率
   - 平均扣分
   - 建议复核人数
2. **错因分布**
   - 计算错误占比
   - 逻辑错误占比
   - 概念错误占比
3. **高频错误样例**
   - 学生作答证据片段
   - 系统解释
   - 可填写教师点评
4. **讲评建议**
   - 推荐讲解点
   - 常见混淆点
   - 易错公式/单位问题

### 商业价值

这一页的价值通常会高于单学生报告，因为它直接服务教师讲评与备课。

---

## 场景 C：异常样本复核

### 教师目标

老师不想重新看所有卷子，只想看那些：

1. 识别不清
2. 机器不确定
3. 高扣分但证据弱
4. 明显异常

### 理想交互

1. 进入待复核列表
2. 默认按优先级排序
3. 点开单份样本
4. 左看原图，中看证据，右看扣分理由
5. 一键改判 / 保留 / 标注为高价值样本

### 推荐优先级规则

从高到低可按：

1. `REJECTED_UNREADABLE`
2. `requires_human_review=true`
3. 低置信度
4. 高扣分且证据薄弱
5. 教师主动筛选的题型

---

## 6. 前端中，教师应该如何输入

你现在的输入方式仍然比较像“接口表单”。  
更合理的方式，是把教师输入拆成四层。

## 6.1 任务级输入

作用：任务管理、历史检索、报告命名，不直接给模型。

推荐字段：

1. 学科
2. 年级
3. 班级/批次名称
4. 作业/试卷名称
5. 题号或题组
6. 总分
7. 单题 / 整卷
8. 是否需要讲评分析

## 6.2 评分规则输入

作用：显式告诉系统如何判，而不是把规则隐含在 prompt 里。

推荐字段：

1. 标准答案文件
2. 使用已有 rubric / 重新生成 rubric
3. 是否允许等价表达
4. 是否允许跳步
5. 是否严格单位
6. 是否严格有效数字 / 舍入
7. 数值容差要求
8. 哪些错误类型要重点标记

### 说明

你已经在 prompt 中显式写了数值容差和舍入规则，这其实说明：

> **未来要把这些规则前置为教师可配置输入，而不是一直藏在 prompt 里。**

## 6.3 复核级输入

作用：修正结果 + 沉淀数据。

推荐字段：

1. 最终判定
2. 最终得分
3. 教师点评
4. 错误归因标签
5. 是否纳入高价值标注资产

## 6.4 导出级输入

作用：决定最终输出给谁看。

推荐字段：

1. 导出学生个人报告
2. 导出班级汇总
3. 导出讲评版
4. 是否显示证据片段
5. 是否显示内部机器字段

---

## 7. 前端应该有哪些关键页面

## 7.1 任务创建向导

建议替代当前“直接上传”的方式，做成 5 步：

1. 选择任务类型
2. 上传参考答案
3. 上传学生作答
4. 配置评分规则
5. 确认提交

### 为什么重要

这一步会直接把系统从“工程台”变成“老师能理解的产品入口”。

## 7.2 任务进度页

当前你已经有状态与 SSE，这一页可以做得很强。

建议展示：

1. 当前状态
2. 已完成人数 / 总人数
3. 当前阶段（预处理 / 感知 / 评分 / 落库）
4. 异常样本数
5. 预计剩余时间

## 7.3 班级结果看板

这是未来最关键的一页之一。

建议展示：

1. 班级通过率
2. 平均扣分
3. 题目正确率
4. 高频错误类型
5. 建议优先复核的学生
6. 讲评建议

## 7.4 学生个人报告页

建议信息结构：

1. 总体结论
2. 得分 / 扣分
3. 扣分点列表
4. 对应证据片段
5. 修改建议
6. 教师复核意见

## 7.5 复核工作台 2.0

建议布局：

1. 左侧：学生信息、任务信息、总分、风险标签
2. 中间：原图 / 作答图像 / 证据定位
3. 右侧：判分理由、错因标签、操作按钮
4. 底部：教师复核输入区

---

## 8. 产出的结果应该长什么样

## 8.1 学生个人报告

推荐结构：

1. **总体结果**
   - 是否通过
   - 最终得分
   - 总评
2. **扣分点**
   - 错误类型
   - 扣分值
   - 为什么扣
   - 如何改
3. **证据片段**
   - 对应原始作答文本或图像片段
4. **教师备注**
   - 如果已复核，显示人工最终意见

## 8.2 教师阅卷视图

和学生报告不应完全一样，应更偏“决策界面”：

1. 任务摘要
2. 学生状态
3. 判分矩阵
4. 证据片段
5. 风险标签
6. 一键复核操作

## 8.3 班级汇总报告

建议至少包含：

1. 班级正确率
2. 平均扣分
3. 题目维度表现
4. 高频错因排行
5. 高频证据样例
6. 建议讲评重点

---

## 9. 商业化建议

## 第一阶段：卖“提效”

核心卖点：

1. 批量阅卷更快
2. 老师只看争议样本
3. 自动生成讲评依据

这时候不要强调“平台”“智能体”“多模态中台”，要强调：

> **替老师省时间。**

## 第二阶段：卖“数据资产”

当复核和标注沉淀起来后，可以卖：

1. 题目级错因分析
2. 班级趋势分析
3. 教研共建 rubric
4. 高价值题库/讲评资产

## 第三阶段：卖“工作流”

那时才适合继续往：

1. 年级组协作
2. 机构级部署
3. 多教师协同
4. 多学科扩展

---

## 10. 未来 1-2 个版本我建议的优先顺序

## 版本重点 A：把输入显式化

优先让教师在前端配置：

1. 等价表达容忍度
2. 单位严格性
3. 数值容差
4. 有效数字/舍入规则
5. 是否允许跳步

## 版本重点 B：把输出从“结果”升级成“可行动信息”

重点不是多显示几个 JSON 字段，而是直接回答：

1. 哪些学生该复核？
2. 哪些错误最常见？
3. 本题讲评建议是什么？

## 版本重点 C：把前端收成教师工作流

优先做：

1. 任务创建向导
2. 班级汇总页
3. 学生报告升级
4. 复核优先队列

---

## 11. 最后的判断

这套系统最有希望做成的，不是“万能教育 AI”，而是：

> **老师在批量阅卷、异常复核、讲评准备中愿意反复使用的 AI 工作台**

只要你把这个定位守住，当前已经写好的很多深能力——异步任务、SSE、报告 API、标注资产、Prompt 控制、runtime telemetry——都会变成真正有商业意义的产品资产，而不是一堆“很强但讲不清卖什么”的技术堆栈。
