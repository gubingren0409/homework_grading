# SQLite -> PostgreSQL 迁移规划（中文）

本文对应 **Phase 7 / P7-03**。目标不是现在立刻迁移，而是回答三个问题：

1. 什么时候该迁
2. 先迁哪些表最值钱
3. 如何分阶段推进

---

## 1. 为什么现在还没必要立刻迁

当前系统的主矛盾仍然是：

- 产品收口
- 教师工作流打磨
- 复核闭环
- 部署治理

不是数据库能力不足本身。

SQLite 现在仍然能支撑：

- 单机部署
- 中低并发
- 小规模试点
- 演示环境

因此当前更合理的策略是：

> **先把 PostgreSQL 当成扩容路线，而不是当前版本的硬切换任务。**

---

## 2. 什么时候应启动迁移

出现以下信号时，就应从“规划”转向“实施”：

1. API / Worker 需要多实例部署
2. SQLite 锁竞争开始频繁影响任务提交或结果落库
3. 运营统计查询开始和主业务写入互相影响
4. 需要更强的备份、恢复、权限控制能力
5. 需要多租户或更严格的数据隔离

### 2.1 量化触发阈值（P9-09）

为避免"感觉到了就迁"的模糊判断，定义以下可观测的硬阈值。**任一条件持续触发 3 天以上即应启动迁移立项**：

| 指标 | 阈值 | 说明 |
| --- | --- | --- |
| 同时处于 PROCESSING 的任务数 | ≥ 5 | 并发写入开始触发 SQLite WAL 锁等待 |
| 单日提交任务数 | ≥ 50 | 日写入量超出 SQLite 舒适区 |
| `database is locked` 错误率 | ≥ 1次/小时 | 已出现真实锁竞争 |
| Worker 实例数 | ≥ 2 | SQLite 不支持多进程安全写入 |
| `grading_results` 行数 | ≥ 100,000 | 查询性能开始下降 |
| 单次 API 请求平均延迟（DB 操作） | ≥ 500ms | 锁等待导致请求变慢 |

### 2.2 监控接入建议

上述阈值可通过以下方式监控：

1. **运维面板（ops_console）**：已集成任务状态统计（`get_task_status_counts`），可直接展示 PROCESSING 并发数。
2. **日志告警**：在 Worker 日志中搜索 `database is locked` 关键词，接入日志告警（如 Loki/ELK）。
3. **定期巡检脚本**：`/api/v1/ops/queue/diagnostics` 返回队列与任务状态分布，可定时采集。
4. **数据库行数**：定期执行 `SELECT COUNT(*) FROM grading_results` 并记录趋势。

## 3. 迁移优先级

| 优先级 | 表/域 | 原因 |
| --- | --- | --- |
| P1 | `tasks` | 所有任务状态、进度、复核状态都依赖它 |
| P1 | `grading_results` | 报告、班级看板、复核工作台都直接读取 |
| P2 | `rubrics` | 题目标准答案与缓存复用依赖它 |
| P2 | `teacher_review_decisions` | 新增的教师复核闭环数据，需要更稳定持久化 |
| P2 | `golden_annotation_assets` | 数据资产沉淀价值高，未来复用会更多 |
| P3 | `task_runtime_telemetry` / prompt ops / metrics | 更偏运营与观察面，可稍后迁移 |

---

## 4. 哪些场景会最先受益

### 4.1 写入并发

受益最大的是：

- 批量批改时 `tasks` + `grading_results` 的并发写入
- 复核与运营统计同时访问数据库

### 4.2 多实例部署

一旦 API / Worker 分别做多实例：

- SQLite 会立刻成为共享写入瓶颈
- PostgreSQL 会是更自然的中心库

### 4.3 运维能力

PostgreSQL 在这些方面更强：

- 在线备份
- 权限控制
- 连接管理
- 可观察性

---

## 5. 推荐迁移路径

### 阶段 1：先做抽象收口

目标：

- 避免业务代码直接依赖 SQLite 细节
- 让数据访问集中收口在 `src/db/`

当前已经做了一部分，但 `client.py` 仍偏大，后续可继续按数据域拆分。

### 阶段 2：优先兼容 SQL 方言差异

重点检查：

- `INSERT ... ON CONFLICT`
- 时间函数
- JSON 字段处理
- 分页 / 聚合查询

目标是先减少“数据库切换会炸”的语句。

### 阶段 3：双环境验证

做法：

1. 保留 SQLite 默认开发模式
2. 增加 PostgreSQL 测试环境
3. 用同一套 API / 任务流回归验证

### 阶段 4：按核心表迁移

顺序建议：

1. `tasks`
2. `grading_results`
3. `rubrics`
4. review / annotation / telemetry

### 阶段 5：切主库

前提：

- 核心查询已通过回归
- 备份恢复流程已演练
- 观测与报警已接好

---

## 6. 当前最务实的建议

如果只看未来两个版本：

1. **现在不做大迁移**
2. **继续保持 SQLite 默认**
3. **把 PostgreSQL 明确写成扩容路线**
4. **优先解决产品和交付问题**

这是对当前仓库现状最稳妥的策略。
