第一部分：通俗版（给当前需要快速恢复全局感的人）
============================================================

一、这个系统现在到底在做什么？

你可以把它理解成一个“自动批改工厂”，分成两台机器：

1）第一台机器（感知层，Qwen）
- 看图，把学生卷子里的文字、公式、图表抄出来，变成结构化数据。

2）第二台机器（认知层，DeepSeek）
- 拿到结构化数据后，按评分标准判断对错、扣分点，给出评语和置信度。

中间还有一位“总调度”（Workflow）：
- 负责把文件拆页、并行处理、聚合结果、异常拦截、最后输出评分报告。


二、我们已经完成到了什么阶段？

真实代码进展已经到 Phase 31+，并且 Phase 35 的核心代码也已经落地（但验证归档还没补齐）。

目前可确认的状态：
- Phase 28：API 与计算解耦（Celery + Redis）已落地
- Phase 29-34：容错、断路器、可观测、存储适配等关键能力已落地
- Phase 31：Claim Check（只在队列里传引用，不传大文件）已落地
- Phase 35：对称布局预处理（REFERENCE / STUDENT_ANSWER 同管线）已落地代码

注意：README里的阶段描述相对落后，不能作为当前真实进度唯一依据。


三、为什么你会“看不懂现在的框架”？

因为系统已经从“单机脚本”进化成“分层+异步+可扩展架构”：

- API层：只负责接收请求和返回 task_id，不做重计算
- Worker层：真正跑AI推理与批改
- Storage层：负责文件落地与跨节点可读（local/s3）
- Orchestration层：串联感知与认知
- Schema层：保证输入输出契约稳定

这不是变复杂，而是“把复杂性分层收纳”了。


四、现在最大的价值点是什么？

1）稳定性明显提升
- 已有测试目录显示样本 `21/24` 成功（r1_nonstream_cache_test）

2）架构风险显著下降
- 队列不再传大对象，避免Redis爆内存
- 有断路器、重试、降级、DLQ（死信队列）

3）Phase 35 的方向是对的
- 你强调的“对称预处理”原则已经在代码中兑现
- 差异由 prompt 吸收，不通过管线分叉吸收

============================================================
第二部分：专业版（给工程推进与架构决策）
============================================================

一、当前架构剖面（Code Reality）

1）API Gateway（FastAPI）
- 文件：src/api/main.py, src/api/routes.py
- 关键职责：
  - 接收上传
  - 调 storage_adapter 存储文件并拿 URI
  - 写入任务状态（PENDING）
  - 下发 Celery 任务（只传引用）
  - 提供状态查询、SSE流式状态、结果分页

2）Async Worker（Celery）
- 文件：src/worker/main.py
- 关键职责：
  - PROCESSING/COMPLETED/FAILED/REJECTED 状态推进
  - 通过 storage.retrieve_files 还原输入
  - 构建 GradingWorkflow 执行核心链路
  - 结果落库 + Redis Pub/Sub 状态广播
  - 失败重试 + DLQ 路由 + 清理临时文件

3）Workflow（Orchestration）
- 文件：src/orchestration/workflow.py
- 关键职责：
  - process_multiple_files 统一输入（含多页）
  - 可选 layout preprocess（feature flag）
  - 并行 perception（asyncio.gather）
  - 空白卷/不可读短路
  - 全局IR聚合
  - 调 cognitive 评估并补充人工复核标记

4）Perception Engine（Qwen）
- 文件：src/perception/engines/qwen_engine.py
- 关键能力：
  - process_image -> PerceptionOutput
  - extract_layout -> LayoutIR（Phase 35）
  - 坐标清洗（内容bbox + layout bbox）
  - key pool + retry + circuit breaker + semaphore

5）Cognitive Engine（DeepSeek）
- 文件：src/cognitive/engines/deepseek_engine.py
- 关键能力：
  - evaluate_logic -> EvaluationReport
  - generate_rubric -> TeacherRubric
  - 流式/非流式策略切换 + 可降级 deepseek-chat
  - 强约束JSON提取与schema验证

6）Storage Adapter（Local/S3）
- 文件：src/core/storage_adapter.py
- 价值：
  - 统一 claim-check 接口，避免节点间文件系统耦合
  - 可切换 local 与 s3 后端


二、数据契约与状态机

1）PerceptionOutput（感知输出）
- readability_status, elements, global_confidence, is_blank, trigger_short_circuit

2）LayoutIR（Phase 35）
- context_type: REFERENCE | STUDENT_ANSWER
- regions: target_id/question_no/region_type/bbox
- image_width/height 由宿主注入，避免信任模型侧值

3）EvaluationReport（认知输出）
- status: SCORED | REJECTED_UNREADABLE
- total_score_deduction, step_evaluations, requires_human_review 等


三、Phase 35 架构判断（结论）

结论：方向正确，且代码已形成“对称预处理 + prompt多态”。

证据链（代码）：
- workflow.run_pipeline -> context_type="STUDENT_ANSWER"
- workflow.generate_rubric_pipeline -> context_type="REFERENCE"
- qwen_engine.extract_layout 使用 context_type 注入提示词
- image_slicer 采用统一数学裁切逻辑，无语义分叉

即：输入异构由 prompt 处理，非由 pipeline 分叉处理。
