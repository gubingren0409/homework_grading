# 生产前检查清单（中文）

本文对应 **Phase 7 / P7-02**，目标不是把系统直接升级成“云原生生产平台”，而是明确：

1. 现在能安全交付到什么规模
2. 当前单点风险在哪
3. 短期应先补哪些收口项

---

## 1. 当前适用规模

当前架构更适合：

- 单校内试点
- 单老师 / 少量老师并发使用
- 单机部署
- 小批量班级作业批改

当前不建议直接承诺：

- 多租户 SaaS
- 大规模横向扩容
- 高并发全天候生产环境

---

## 2. 生产前最小清单

### 2.1 反向代理

建议在 API 前放一层 **Nginx / Caddy**：

- 统一域名入口
- 静态 TLS 终止
- 限制大请求与超时
- 统一访问日志

### 2.2 HTTPS

如果有对外访问：

- 必须上 HTTPS
- 证书终止建议放在 Nginx / Caddy
- API 容器本身不必直接处理证书

### 2.3 日志入口

最小要求：

- API 日志可查
- Worker 日志可查
- Redis 日志可查

建议至少做到：

1. 容器日志保留
2. 错误日志可按 `task_id` 检索
3. 重要任务链路能从 API 查到 Worker

### 2.4 数据库备份

当前默认 SQLite，因此必须明确：

- 备份文件在哪
- 谁负责备份
- 备份频率
- 恢复步骤

最短期可执行方案：

1. 每天固定时间备份 `outputs/grading_database.db`
2. 备份前优先停写或在低峰执行
3. 备份文件放到另一块磁盘或对象存储

### 2.5 Redis 单点说明

当前 Redis 同时承担：

- Celery broker
- SSE / PubSub 状态流

因此 Redis 挂掉会影响：

- 新任务分发
- 状态流推送
- 实时进度体验

虽然系统存在**本地回退**，但 Redis 仍是当前关键单点。

### 2.6 SQLite 适用边界

SQLite 当前是合理默认值，但边界必须说明：

- 适合单机
- 适合中低并发
- 不适合多实例共享写入
- 不适合复杂运营分析查询

---

## 3. 当前单点风险

| 单点 | 风险 | 当前建议 |
| --- | --- | --- |
| Redis | 队列与状态流都依赖它 | 单机试点可接受，但要明确故障影响 |
| SQLite | 单文件数据库 | 适合当前阶段，但要强制备份 |
| 模型上游 API | 外部依赖 | 需要外呼可达性与 Key 健康检查 |
| 本地磁盘 `outputs/` | 结果、数据库、导出都在这里 | 定期备份，不要把它当临时目录 |

---

## 4. 可执行的短期方案

如果你要在最近一个版本进入“可稳定演示 / 可试点交付”状态，建议短期只做这些：

1. 加 Nginx
2. 上 HTTPS
3. 明确 `.env` 配置流程
4. 明确 SQLite 备份流程
5. 明确 Redis 故障后的降级说明
6. 把部署与排障文档固定下来

这是一条**低风险、真实可执行**的路线。

---

## 5. 当前缺口列表

仍未完全收口的项：

1. 没有完整的外部错误追踪平台接入说明
2. 没有正式的数据库迁移框架
3. 没有 Redis 高可用方案
4. 没有对象存储替代本地文件路径
5. 没有真正面向生产的多实例部署方案

---

## 6. 我建议你现在怎么描述它

更准确的说法是：

> **当前系统已经适合单机、小规模试点和正式演示，但仍处在“可交付原型 / 试点版”阶段，不应直接包装成大规模生产 SaaS。**

这比“已经生产可用”更诚实，也更利于后续迭代。
